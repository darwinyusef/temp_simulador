1 · Descripción general

El laboratorio simula la configuración completa de un switch Cisco Catalyst de capa 2 en el entorno hospitalario Clínica San Rafael. El estudiante construye la topología desde cero arrastrando dispositivos al canvas libre (drag-and-drop) e interactúa con una terminal Cisco IOS simulada para completar cinco pasos progresivos que cubren los fundamentos del switching L2.

Escenario

El SW-Core de la clínica conecta las PCs de médicos y RRHH con el servidor de expedientes. El ingeniero de red debe configurar VLANs, trunks, STP y port security antes de la puesta en producción.

Tecnologías simuladas

  • Cisco IOS 15.2 (CLI interactivo)
  • IEEE 802.1Q trunk encapsulation
  • IEEE 802.1w RSTP
  • IEEE 802.1D STP
  • Cisco Port Security

2 · Objetivos de aprendizaje

Al finalizar el laboratorio, el estudiante será capaz de:

Conceptuales

  • Explicar el proceso de MAC address learning y flooding
  • Describir la estructura de la tabla CAM y su TTL
  • Distinguir entre VLANs de acceso y trunks 802.1Q
  • Explicar la elección de Root Bridge en STP/RSTP
  • Identificar los vectores de ataque que mitiga Port Security

Procedimentales

  • Crear y nombrar VLANs en Cisco IOS
  • Asignar puertos de acceso a VLANs
  • Configurar un puerto trunk con VLANs permitidas
  • Activar RSTP y verificar puertos bloqueados
  • Configurar port-security con sticky MACs y modo violación
🎯
Conexión curricular: Este lab consolida el Lab 1 (VLANs drag-and-drop) y Lab 2 (Router-on-a-Stick) con la perspectiva de la configuración del switch como dispositivo central. Es prerrequisito recomendado para los laboratorios de ciberseguridad activa (Lab 4: MAC Flooding, Lab 5: ARP Spoofing).

4 · Guía paso a paso

1
MAC Address Learning — Tabla CAM
Automatico 15 pts ~5 min

Qué hace el estudiante

Arrastra desde el panel lateral los dispositivos necesarios: 1 SW-Core, 1 Switch L2, al menos 2 PCs o Laptops y opcionalmente un Servidor. Luego abre el modal ⚡ Configurar y hace clic en Enviar tramas de red. El simulador anima el proceso de MAC learning enviando tramas desde cada host al SW-Core. La tabla CAM se construye progresivamente.

Puntos clave para reforzar en clase

  • La primera trama siempre genera flooding — el switch aún no sabe dónde está el destino
  • El flooding es por segmento, no por VLAN (las VLANs se configuran en pasos posteriores)
  • La tabla CAM tiene TTL de ~300 s; en el simulador persiste para el ejercicio
  • Las entradas Dynamic se aprenden; las Static se configuran manualmente

Pregunta de verificación rápida

❓

¿Qué pasaría si dos hosts tuviesen la misma dirección MAC en la misma VLAN? → El switch sobrescribiría la entrada CAM y podría causar pérdida de tráfico o un loop de aprendizaje.

Error común

El estudiante pasa al paso 2 sin ver la tabla CAM. Recordarle abrir el modal 📋 MAC Table para visualizar las entradas antes de avanzar.

2
VLANs — Segmentación L2
CLI IOS 20 pts ~15 min

Secuencia correcta de comandos

-- Crear VLANs --
SW-Core> enable
SW-Core# configure terminal
SW-Core(config)# vlan 10
SW-Core(config-vlan)# name Medicos
SW-Core(config-vlan)# exit
SW-Core(config)# vlan 20
SW-Core(config-vlan)# name Administracion
SW-Core(config-vlan)# exit
SW-Core(config)# vlan 30
SW-Core(config-vlan)# name Servidores
SW-Core(config-vlan)# exit

-- Asignar puertos de acceso --
SW-Core(config)# interface Fa0/1
SW-Core(config-if)# switchport mode access
SW-Core(config-if)# switchport access vlan 10
SW-Core(config-if)# exit
SW-Core(config)# interface Fa0/2
SW-Core(config-if)# switchport mode access
SW-Core(config-if)# switchport access vlan 20
SW-Core(config-if)# exit
SW-Core(config)# interface Fa0/4
SW-Core(config-if)# switchport mode access
SW-Core(config-if)# switchport access vlan 30

-- Verificación --
SW-Core# show vlan brief

Unlock trigger

El lab desbloquea el paso 3 cuando las tres VLANs tienen nombre personalizado (Medicos, Administracion, Servidores). La asignación de puertos es parte del aprendizaje pero no es requisito para desbloquear.

⚠️
Error frecuente #1: El estudiante escribe name Médicos con tilde. La CLI lo acepta pero el simulador espera exactamente Medicos (sin tilde). Indicar que en IOS real los caracteres especiales pueden causar problemas de interoperabilidad.
Error frecuente #2: Intentar switchport access vlan 10 antes de crear la VLAN. El simulador muestra error: "VLAN not in database".

Concepto para reforzar

Mostrar en el canvas cómo las insignias de VLAN sobre los nodos cambian de color al asignar los puertos. Preguntar: ¿pueden PC-Admin (VLAN 10) y PC-RRHH (VLAN 20) hacerse ping directamente? No — necesitan routing L3.

3
Trunk 802.1Q — Uplink etiquetado
CLI IOS 20 pts ~10 min

Secuencia correcta

SW-Core# configure terminal
SW-Core(config)# interface Gi0/1
SW-Core(config-if)# switchport mode trunk
SW-Core(config-if)# switchport trunk allowed vlan 10,20,30
SW-Core(config-if)# exit

-- Verificación --
SW-Core# show interfaces trunk

Visualización en el canvas

Al configurar el trunk con la herramienta 🔀 Trunk 802.1Q del canvas o vía CLI, el enlace entre SW-Core y SW-2 cambia de apariencia y aparece la etiqueta del trunk. Aprovechar este momento visual para explicar el encabezado 802.1Q.

Diagrama conceptual para pizarra

[ Ethernet Header ][ 802.1Q Tag ][ Payload ][ FCS ]
                     ├─ TPID: 0x8100 (2B)
                     ├─ PCP:  3 bits (QoS)
                     ├─ DEI:  1 bit
                     └─ VID:  12 bits (0–4094)

VLAN nativa (default: VLAN 1): tráfico SIN etiqueta.
Riesgo: VLAN hopping si la VLAN nativa no está segregada.
💡
Pregunta de discusión: ¿Por qué configuramos trunk allowed vlan 10,20,30 en vez de dejar todas las VLANs? → Principio de mínimo privilegio: solo las VLANs necesarias reducen el dominio de broadcast y limitan el radio de daño ante un misconfiguration.
4
STP / RSTP — Prevención de loops L2
CLI IOS 20 pts ~15 min

Secuencia correcta

SW-Core# configure terminal
SW-Core(config)# spanning-tree mode rapid-pvst
SW-Core(config)# end

-- Verificación (CRÍTICO para unlock) --
SW-Core# show spanning-tree vlan 10

-- Salida esperada --
VLAN0010
  Spanning tree enabled protocol rstp
  Root ID  Priority 24586  This bridge is the root
  ...
  Interface  Role  Sts   Cost
  Gi0/3      Altn  BLK   4    ← enlace redundante bloqueado

Visualización en el canvas

Al ejecutar spanning-tree mode rapid-pvst y el show de verificación, el simulador registra el estado STP. El modal 🌳 STP muestra el estado de cada puerto, incluyendo cuál enlace quedaría bloqueado en una topología redundante. Preguntar: ¿qué pasaría si hay dos caminos entre SW-Core y SW-2?

Conceptos a desarrollar en clase

ConceptoExplicación prácticaObservación en el lab
Root Bridge El SW-Core gana porque tiene el menor Bridge ID (prioridad 32768 + MAC más baja) This bridge is the root en show spanning-tree
Root Port Puerto de cada switch no-root con menor costo al Root No visible (SW-Core es Root)
Blocking Gi0/3 queda en Alternate → no reenvía tramas de datos ⊗ BLK en el canvas · Gi0/3 en rojo
RSTP vs STP RSTP converge en <1s; STP clásico necesita 30–50s Protocol: rstp en la salida show
🔴
Concepto crítico: Un loop L2 sin STP genera un broadcast storm. Una sola trama de broadcast se replica infinitamente, saturando toda la red en milisegundos. Demostrar con el modal STP sin RSTP activo (status: ⚠ loop!).

Error frecuente

El estudiante ejecuta spanning-tree mode rapid-pvst pero no corre show spanning-tree vlan 10. El simulador requiere el show como verificación explícita para hacer unlock — igual que en un entorno real donde verificar es parte de la configuración.

5
Port Security — Control de acceso L2
CLI IOS 25 pts ~15 min

Secuencia correcta

SW-Core# configure terminal
SW-Core(config)# interface Fa0/1
SW-Core(config-if)# switchport mode access         ← debe ser access, no trunk
SW-Core(config-if)# switchport port-security
SW-Core(config-if)# switchport port-security maximum 1
SW-Core(config-if)# switchport port-security violation restrict
SW-Core(config-if)# end

-- Verificación --
SW-Core# show port-security
Fa0/1    1       1        0           restrict

Unlock trigger

El simulador desbloquea cuando los tres comandos de port-security están configurados: port-security (enable), maximum y violation. Al completarse, el modal 🔒 Port Sec muestra el estado de violación simulada.

Tabla de modos de violación

ModoDescarta tráficoContadorSyslogAcción en puerto
protect✓✗✗Sigue activo (silencioso)
restrict✓✓✓Sigue activo
shutdown default✓✓✓err-disabled → requiere shutdown/no shutdown
🔗
Conexión con Lab 4 (MAC Flooding): Port Security es la defensa directa contra el ataque MAC Flooding que se explora en Lab 4. Si el estudiante ya completó Lab 4, preguntar: ¿cuántas MACs necesitaría inundar el atacante con maximum 1 activo? → Solo 1 MAC extra para desencadenar la violación.

6 · Preguntas de discusión

Durante el laboratorio

  1. ¿Por qué el primer envío de una trama genera flooding y el segundo no?
  2. ¿Qué problema resuelve la segmentación en VLANs respecto a los dominios de broadcast?
  3. Si PC-Admin intenta hacer ping a Srv-Med (VLAN 30), ¿el switch lo reenvía directamente? ¿Por qué?
  4. ¿Qué diferencia hay entre un puerto en modo access y uno en trunk?
  5. ¿Por qué el enlace Gi0/3 hacia SW-3 se bloquea y no otro enlace?

Post-laboratorio / evaluación oral

  1. Un hospital tiene 500 dispositivos IoT médicos. ¿En qué VLAN los pondrías y por qué los aislarías?
  2. ¿Cómo afecta una entrada MAC inválida en la tabla CAM a la seguridad de la red?
  3. Si el Root Bridge (SW-Core) falla, ¿qué ocurre con el enlace Gi0/3 en BLK?
  4. ¿Cuál modo de violación de Port Security usarías en una UCI donde no puedes desconectar equipos de soporte vital? Justifica.
  5. ¿Qué es el VLAN hopping attack y qué configuración de este lab lo previene?

9 · Glosario técnico de referencia

Tabla CAM
Content-Addressable Memory. Base de datos del switch que mapea MAC → Puerto → VLAN. Búsqueda O(1) gracias a la memoria asociativa.
VLAN (IEEE 802.1Q)
Virtual LAN. Segmentación lógica L2 que crea dominios de broadcast independientes sobre la misma infraestructura física.
Trunk Port
Puerto configurado para transportar múltiples VLANs etiquetadas con cabecera 802.1Q. Usualmente conecta switches entre sí o con routers.
VLAN Nativa
VLAN cuyos frames no llevan etiqueta 802.1Q en el trunk. Por defecto VLAN 1. Vector de VLAN hopping si no se configura correctamente.
STP / RSTP
Spanning Tree Protocol (802.1D) / Rapid STP (802.1w). Protocolo L2 que previene loops bloqueando puertos redundantes. RSTP converge en <1s.
Root Bridge
Switch elegido como raíz del árbol STP. Todos los cálculos de costo de ruta parten de él. Elegido por menor Bridge ID (prioridad + MAC).
Port Security
Feature Cisco que limita MACs por puerto. Mitiga MAC flooding y acceso físico no autorizado. Modos: protect, restrict, shutdown.
err-disabled
Estado en que el switch deshabilita automáticamente un puerto por violación de seguridad u otro evento. Requiere shutdown + no shutdown para recuperar.
